### Task 3
- 採用方針: `local-watcher/bin/modules/adjudicator.sh` を新規作成（200 行弱）。`adj_gate_enabled` は既に正規化済みの `PR_REVIEWER_ADJUDICATOR_ENABLED` を厳密 `=true` で評価する 5 行関数（重複正規化なし / Req 5.1）。`adj_extract_findings` は awk で `## 指摘事項` 配下 bullet 行を section 境界（次 `## ` 見出し / EOF）で正しく区切り、行頭 `- [(high|medium|low)] <file>:<line> — <内容>` の厳密 regex で parse して TSV 出力、jq で JSON 配列化。reconciliation check は bullet 総数 vs parse 件数を突合し、不一致時のみ `adj_warn` で WARN + rc=4（design.md Components and Interfaces 節の関数 contract そのまま）。`REQUIRED_MODULES` 配列の `"pr-reviewer.sh"` 直後に `"adjudicator.sh"` を追記（issue-watcher.sh:1329）。
- 重要な判断: (1) **awk 単一パスで 2 系統出力**: bullet 総数（reconciliation 用）と parse 成功 TSV を同じ awk スクリプトで 1 パス算出する設計を採用（review_text を 2 回 awk に流すより O(N) で済む）。awk 末尾 `END { printf "BULLET_TOTAL=%d\n", bullet_total }` で総数を尻に付け、bash 側で `grep -E '^BULLET_TOTAL='` と `grep -E '^PARSED\b'` で分離。(2) **section 境界の awk 状態機械**: `in_section = 1` を `^## 指摘事項[[:space:]]*$` で立て、`in_section && /^## /` で他の `## ` 見出しが来たら降ろす。これにより `## 結論` 後の bullet が誤って混入しないことを ケース 5 で確認。(3) **file:line 抽出の右端 colon 検索**: `<file>:<line>` の file 名にも path separator (`/`) が含まれ得るが `:` は含まない前提で、awk で末尾から `:` を走査し file / line に分割。line は `[0-9]+` 正規表現で再検証してから採用（防御的書式 drift 対応）。(4) **jq への未信頼入力渡し**: codex 出力（PR コメント由来 = 未信頼）は jq に `--arg` / `--argjson` で literal 渡し（CLAUDE.md §5 整合 / filter inline 展開禁止）。line は数値のため `--argjson` で int 型保持。(5) **`adj_warn` の遅延束縛**: adjudicator.sh は `adj_warn` を呼ぶが定義は core_utils.sh 側にある。bash の遅延束縛により呼び出し時に解決され、本モジュールは再定義しない（CLAUDE.md「機能追加ガイドライン §2」整合）。テスト側は `extract_function` で関数 1 つだけ隔離するため、test 冒頭で `adj_warn` を stub で潰す（fail-safe）。(6) **「指摘なし」プレーン行のみは reconcile 対象外**: codex の `指摘なし` テキストは bullet 行ではなく、awk の bullet カウントが 0 のため `bullet_total -gt 0` ガードで自然に rc=0 に倒れる（design.md の `[]` 返却契約と整合）。
- 残存課題: なし。本 task 終了時点で adjudicator.sh は REQUIRED_MODULES に登録済みで `source` 可能だが、`adj_run_for_pr` を呼ぶ配線（pr-reviewer.sh 末尾 hook）は task 6 のスコープのため、現状の watcher 起動経路では adjudicator 関数群は呼ばれない（NFR 2.1 既定 OFF 完全 no-op が維持される）。検証コマンドと結果: `bash -n local-watcher/bin/modules/adjudicator.sh` OK / `shellcheck local-watcher/bin/*.sh local-watcher/bin/modules/*.sh` 警告ゼロ / `bash local-watcher/test/adj_resolve_gate_test.sh` PASS=8 FAIL=0 / `bash local-watcher/test/adj_extract_findings_test.sh` PASS=23 FAIL=0 / 既存テスト 3 種 退行ゼロ（pr_publish_commit_status PASS=74 / pr_publish_claude_status_from_branch PASS=52 / pr_default_prompt PASS=52） / `diff -r .claude/agents repo-template/.claude/agents` / `diff -r .claude/rules repo-template/.claude/rules` 差分ゼロ。

### Task 4
- 採用方針: `local-watcher/bin/modules/adjudicator.sh` に `adj_classify_findings`（約 180 行）と `adj_validate_decisions`（約 70 行）を追記。`adj_classify_findings` は (i) `findings_json == []` / 空文字を早期 return（claude 起動しない / Req 1.4 観点で副作用ゼロ）、(ii) PR 番号 / SHA / base / head に shell metachar 検査 + 数値・hex 形式の strict 検証（pr_substitute_placeholders 流の防御）、(iii) `$HOME/bin/adjudicator-prompt.tmpl` を読みつつ `PR_REVIEWER_ADJUDICATOR_PROMPT` env で本文 override 可能（pr-reviewer.sh の `PR_REVIEWER_PROMPT` と同 semantics）、(iv) bash パラメータ展開で 7 placeholder（`{PR}` / `{SHA}` / `{BASE}` / `{HEAD}` / `{REVIEW_TEXT}` / `{SPEC_DIR}` / `{REQUIREMENTS_MD}`）を置換、(v) mktemp で prompt / out / err の 3 tempfile を作り RETURN trap で削除、(vi) `timeout claude -p ... --output-format json --permission-mode plan --model ...` を直接 invoke（`eval` 不使用 / `--permission-mode plan` で Bash/Edit/Write を Claude 側で構造的にブロック / design.md Security Considerations 第 4 項 = bypassPermissions 不使用と整合）、(vii) 実行直後 `git status --porcelain` で workspace-modified 検出時は `git checkout -- .` + rc=3、(viii) `jq -r '.result // empty'` で SDK 標準出力 `{"type":"result",...,"result":"..."}` から本文を抜き、awk で `{...}` を雑に切り出してから `jq -e '.'` で valid JSON 検証 + stdout 出力。`adj_validate_decisions` は decisions_json が JSON object かつ `.decisions` array / `.summary` object であることに加え、件数一致 / verdict ∈ {legitimate, excessive} / id が 1〜N 連番 / summary 集計 vs decisions verdict 集計の一致を jq filter で連続検証する。
- 重要な判断: (1) **引数構成の拡張（4 → 6 引数）**: design.md interface 表は厳密 4 引数だが、`{BASE}` / `{HEAD}` placeholder を tmpl に流し込むため `base_ref` / `head_ref` を引数 5,6 として追加した。pr-reviewer.sh の `pr_execute_review_command` でも同様の契約拡張の前例（design 表より引数多い）が impl-notes に記録されており、本 task でも同じ正当化を行う。(2) **`{REVIEW_TEXT}` 流し込み方式 = 候補 B（findings_json 再構成）**: prompt 文字列構築のための呼び出し元拡張（review_text 全文を引き継ぐ案 A）と、findings_json から `## 指摘事項\n- [sev] file:line — message\n...` を再構成する案 B のうち、**B を採用**。理由は (a) design.md interface 表 4 引数の拡張幅を最小化（review_text を持ち回らない）、(b) stub claude テストで findings_json のみで擬似実行可能（テスト容易性）、(c) adjudicator-prompt.tmpl の指示は「`## 指摘事項` 配下 bullet 行を登場順に id=1, 2, 3 と独立採番」であり、findings_json の順序保持と整合する。trade-off として codex の `## 概要` / `## 結論` 散文が prompt に渡らないが、adjudicator の判定根拠は bullet 行（指摘内容）と requirements.md（AC）であり、散文の欠落は分類精度に影響しない。(3) **`--permission-mode plan` 採用**: design.md Security Considerations 第 4 項に従い `bypassPermissions` を使わず、Claude 側で Bash / Edit / Write を構造的にブロックする `plan` モードを採用。これは tmpl 本文の read-only 制約宣言（Bash / Edit / Write を使わない）の defense-in-depth として機能する。(4) **`eval` 不使用**: security-review.sh の `SECURITY_REVIEW_CLAUDE_CMD` 経路は運用者 override 用 cmd template の bash -c 化に `eval` を回避しているが、adjudicator では tmpl の cmd 化 env が存在しないため `claude` を直接呼ぶ単純経路にした（CLAUDE.md「禁止事項」と整合）。(5) **claude 出力からの JSON 抽出 awk 関数**: `claude --output-format json` の `.result` フィールドは生 JSON 文字列を期待するが、tmpl 指示違反で前置き散文や code fence が混入する case に備え、最初の `{` から depth=0 で閉じる `}` までを awk で抜き出す防御層を追加した。これがケース 4（散文のみ）で rc=2 への倒し込みを担保する。(6) **`adj_validate_decisions` の id 連番 sanity check**: adjudicator-prompt.tmpl の id 採番規約「登場順に 1, 2, 3 と独立採番」と整合するため、`sort` + `range(1; N+1)` 比較で連番を検証。Claude が id を 0 始まり / 飛び番号で返す逸脱を早期検出できる。(7) **stub claude / stub git / stub timeout の PATH 注入**: 本番コードに「テスト専用の env override 経路」を追加しない方針（method A）を採用。tempdir に stub `claude` / `git` / `timeout` を chmod 755 で配置し、test 冒頭で `PATH="$TMP_BIN:$PATH"` する。`timeout` も stub する理由は test 環境に GNU `timeout` が無いケースを避けるため。`git` stub は read-only invariant 検査の `git status --porcelain` を常に「clean」に返すよう no-op 化。(8) **stub claude が `--output-format json` の SDK 標準形式を返す**: `{"type":"result","subtype":"success","result":"<embedded JSON 文字列>"}` を jq で組み立て、`.result` フィールドに本来 Claude が生成する adjudicator JSON 本文を文字列として埋め込む。これにより adj_classify_findings 内の `jq -r '.result // empty'` 経路がテストで正常動作する。(9) **fallback モード（`PR_REVIEWER_ADJUDICATOR_FALLBACK_ON_FAIL`）は本 task で参照しない**: 関数 contract は rc を返すだけで、`passthrough` / `legitimate` の選択は呼び出し元 `adj_run_for_pr`（task 6 スコープ）の責務。テストではこの前提を踏まえて 4 種 rc（0=ok / 1=exec 失敗 / 2=JSON parse 失敗 / 3=workspace-modified）が明示的に分離されることを assert する。
